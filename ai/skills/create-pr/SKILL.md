# Create PR

Create a GitHub pull request using `gh`, auto-detecting and filling any PR template.

## Arguments (parsed from user input)

- `--draft` — create the PR as a draft
- `<title>` — optional title hint (if omitted, derive from commits)

Example invocations:

- `/create-pr`
- `/create-pr --draft`
- `/create-pr Add support for webhook retries`
- `/create-pr --draft Fix race condition in job queue`

## Your Task

### Step 1: Parse Arguments

From the user input, extract:

- `draft` = true if `--draft` is present
- `title_hint` = remaining text after removing `--draft`, or empty string

### Step 2: Get Git Context

Run these in parallel:

```bash
git rev-parse --abbrev-ref HEAD          # current branch name
git remote show origin | grep 'HEAD branch' | awk '{print $NF}'  # base branch (main/master)
git log main..HEAD --oneline             # commits on this branch (replace main with actual base)
git diff main..HEAD                      # full diff vs base (replace main with actual base)
```

Store: `branch`, `base`, `commits` (list), `diff`.

Also fetch the existing PR if one already exists:

```bash
gh pr view --json number,title,body,isDraft,url 2>/dev/null
```

If a PR already exists, note its URL — you will **update** it, not create a new one (see Step 6).

### Step 3: Find PR Template

Search for a PR template in this order. Stop at the first match:

1. `.github/pull_request_template.md`
2. `.github/PULL_REQUEST_TEMPLATE.md`
3. `.github/PULL_REQUEST_TEMPLATE/` — if directory exists, list files and pick the most general one (e.g., `default.md` or the only file); if multiple non-obvious options, ask the user which to use
4. `docs/pull_request_template.md`
5. `pull_request_template.md`

Store the template content in `template` (empty string if none found).

### Step 4: Compose PR Title and Body

#### Title

- If `title_hint` is non-empty, use it directly
- Otherwise, derive a short imperative title from the commits (e.g., "Add webhook retry support")
- Keep it under 70 characters

#### Body

Fill in the template if one was found. For each section:

- Read the section heading and any placeholder text
- Fill it in based on the commits and diff
- **Describe final state, not the journey** — say what the code does now, not what it replaced or how it evolved
- Remove unfilled optional sections rather than leaving placeholder text
- Leave checkboxes intact; check ones that are clearly satisfied by the diff

If no template was found, write a clean description:

- 1–3 bullet points summarizing what the PR does
- A short **Test plan** section describing how to verify

#### Style rules

- Write as the repository author — no AI attribution, no "generated by" language
- Use actual ellipsis (…) not three dots (...)
- Comments and descriptions are concise prose with proper grammar

### Step 5: Show Preview and Confirm

Display the proposed title and body to the user:

```
Title: <title>

---
<body>
---

Create PR? [draft: yes/no] (y to confirm, e to edit, q to quit)
```

Wait for user confirmation before proceeding. If the user wants edits, apply them and show the preview again.

### Step 6: Create or Update PR

#### Creating a new PR

```bash
gh pr create \
  --title "<title>" \
  [--draft] \
  --body "$(cat <<'EOF'
<body>
EOF
)"
```

Use a heredoc to avoid shell quoting issues with the body.

#### Updating an existing PR

If a PR already existed (found in Step 2), update it instead:

```bash
gh pr edit <number> \
  --title "<title>" \
  --body "$(cat <<'EOF'
<body>
EOF
)"
```

If `draft` flag was passed and the PR is not already a draft, also run:

```bash
gh pr ready --undo <number>
```

### Step 7: Return Result

Display the PR URL on success. If the command fails, show the full error output and stop — do not retry silently.
